# Аудит репозитория и приоритетные улучшения (2026-04-23)

## Коротко
Репозиторий уже хорошо организован для персонального workflow, но есть 4 зоны роста с максимальным эффектом:
1. **Надёжность daily-run** (устойчивость к частичным сбоям API и дедупликация повторных запусков).
2. **Качество данных** (валидация схемы снапшотов и контроль аномалий).
3. **Операционка и наблюдаемость** (чёткий лог статусов шага и итоговый health-check).
4. **Сопровождаемость** (типизация, CLI-параметры, унификация конфигов).

---

## Что улучшить в первую очередь

### 1) Идемпотентность и защита от дубликатов
**Проблема:** при повторном nightly-run можно повторно отправить отчёт в Telegram и/или перезатереть артефакты без маркера версии.

**Улучшение:**
- Добавить `run_id` и `generated_at` в `data/reports/*.md` и `data/snapshots/*.json`.
- Перед отправкой в Telegram проверять: не отправлялся ли уже отчёт для `TARGET_DATE`.
- В commit message фиксировать режим (`daily`, `rerun`, `manual-fix`).

**Польза:** меньше риска спама и понятный аудит повторных прогонов.

### 2) Валидация входных/выходных данных
**Проблема:** структура сырых данных и снапшотов подразумевается неявно; ошибки формата всплывают поздно.

**Улучшение:**
- Ввести JSON Schema для:
  - `data/raw/intervals/YYYY-MM-DD.json`
  - `data/raw/garmin/YYYY-MM-DD.json`
  - `data/snapshots/YYYY-MM-DD.json`
- Проверять схему в `normalize.py` до и после трансформации.
- Добавить smoke-test на «битые» fixture-файлы.

**Польза:** ранний фейл с понятной причиной и меньше тихих ошибок в аналитике.

### 3) Явный CLI вместо неявных env-only сценариев
**Проблема:** сейчас часть поведения захардкожена (например, `yesterday`), что усложняет backfill и отладку.

**Улучшение:**
- Добавить аргументы:
  - `fetch_intervals.py --date YYYY-MM-DD --athlete-id ...`
  - `normalize.py --date YYYY-MM-DD --strict`
  - `send_telegram.py --file path --dry-run`
- Сохранить env как fallback, но не как единственный канал конфигурации.

**Польза:** проще ручные прогоны, воспроизводимость и интеграция в CI.

### 4) Наблюдаемость пайплайна
**Проблема:** статус шага виден только в stderr, нет единого «сводного» результата запуска.

**Улучшение:**
- Писать `data/logs/runs/YYYY-MM-DD.json` с полями: `steps`, `duration_ms`, `status`, `errors[]`.
- Добавить мини health-check команду (`python scripts/doctor.py`), которая проверяет:
  - наличие ключевых файлов за дату,
  - валидность JSON,
  - базовую целостность метрик (например, `sleep.total_h` не отрицательный).

**Польза:** проще понять причину сбоя без ручной диагностики.

## Улучшения среднего приоритета

### 5) Типизация и чистые модели
- Вынести модели снапшота в `dataclasses`/`pydantic`.
- Добавить `mypy` (или `pyright`) для критичных скриптов.
- Убрать разрозненные словари в пользу типизированных объектов.

### 6) Тесты на edge-cases
- Тест на дробные/неровные километровые сплиты.
- Тест на отсутствие части Garmin streams.
- Тест на 5xx + backoff в `fetch_intervals.py`.
- Тест на чанкинг Telegram (длина > 4096 и границы UTF-8).

### 7) Документация и onboarding
- Добавить в README:
  - переменные окружения,
  - примеры ручного запуска,
  - «что делать при падении шага N».
- Создать короткий `docs/runbook.md` для аварийных сценариев.
