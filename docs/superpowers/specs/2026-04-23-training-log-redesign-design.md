## 1. Контекст и проблема

Репозиторий `training-log` был заточен под **ночную автоматическую рутину**: скрипт тянул данные из intervals.icu + Garmin (GetFast MCP), собирал дневной отчёт в Telegram, пушил коммит. Код и промпты (`prompts/daily-routine.md`) оптимизированы именно под этот флоу — Claude запускается в 00:00–02:00, анализирует вчера, формирует план на сегодня.

За 4 недели использования (апрель 2026) стало ясно, что реальный паттерн работы другой:

- Влад хочет смотреть в репу **как в тренировочный проект**, а не как в дневник.
- Главный запрос — **текущая неделя** в одном файле: план, статусы, факты, метрики, review, вопросы.
- Нагрузка анализируется **не каждый день**, а раз в несколько дней или по воскресеньям.
- Дневные отчёты в Telegram воспринимаются как «шум по расписанию» — когда открыл TG утром, уже забыл что там было вчера.
- Текущий `data/gym/log.md` организован **по упражнениям** (таблица по болгарскому, таблица по РДЛ…), а Владу нужно мыслить **по сессиям** («Пн, неделя 3, Зал A — что я поднимал»).
- Нет явной иерархии **цели → период → неделя** с review-циклом.
- Нет **README-входа**: открыл репу — не понятно что смотреть первым.

## 2. Ключевые решения

| # | Решение | Выбор |
|---|---------|:-----:|
| 1 | Главный вход в репу | Гибрид: README = цели-ссылки + инлайн текущая неделя |
| 2 | Где живут детали сессий | Отдельные тематические логи (`training/gym.md`, `training/running.md`), организованные по неделям → сессиям |
| 3 | Иерархия планов | Два уровня: `goals.md` (сезон + долгосрок + текущий месяц) + `weeks/YYYY-WW.md` |
| 4 | Роль «рутины» | Ручной запуск; пайплайн догоняет данные до сегодня, обновляет живые файлы, шлёт TG-дайджест |
| 5 | Telegram | Остаётся. Только как дайджест по итогу прогона, не ночной пинг |
| 6 | Анатомия недельного файла | План+статус+сводка+прогрессия+review+open questions |
| 7 | Команды | Две slash-команды: `/training:sync` и `/training:weekly` |
| 8 | Миграция | Полная перезапись старых планов и `data/gym/log.md` в новый формат |
| 9 | Расположение справочников (profile) | Плоский root: `me.md`, `goals.md`, `gym-program.md`, `weight-protocol.md` |

## 5. Slash-команды и пайплайн

### 5.1 `/training:sync` — догнать данные

**Назначение:** один из двух операционных входов. Запускается в любой будний день, пока идёт неделя.

**Что делает:**

1. Определяет `LAST_SYNCED_DATE` = максимальная дата в `data/snapshots/*.json`. Если репа свежая, берёт начало текущей недели.
2. Для каждой даты D в `[LAST_SYNCED_DATE + 1 … today]`:
   - `python scripts/fetch_intervals.py --date D` → `data/raw/intervals/D.json`
   - MCP-вызовы `context_get_recent_sleep`, `context_get_hrv_details`, `context_get_heart_rate` для D → `data/raw/garmin/D.json`
   - Для каждой беговой активности D: `context_list_activities` + `context_get_activity_streams`
   - `python scripts/normalize.py --date D` → `data/snapshots/D.json`
3. Определяет текущую неделю (ISO) → файл `weeks/YYYY-WW.md` (создаёт если нет, используя шаблон из §4.2).
4. Для каждой даты D обновляет строку таблицы «План и статус»:
   - Статус: ✅ если сделано и совпадает с планом, ⚠️ если частично/изменено, ❌ если пропущено без замены
   - Факт: короткая сводка (бег: «10.2 км @ 5:25, HR 132»; зал: «1:50, ЧСС 109/153, kg 12500»)
5. Для каждой новой зальной/беговой сессии добавляет заголовок в `training/gym.md` или `training/running.md` с метриками, **оставляя место для ручного заполнения деталей**.
6. Пересчитывает и перезаписывает секцию «Сводка» в `weeks/YYYY-WW.md`.
7. Формирует Telegram-дайджест (см. §6) и шлёт через `scripts/send_telegram.py`.
8. `git add .` → `git commit -m "sync: YYYY-MM-DD (+N days)"` → `git push`.

**Аргументы:**
- `--from YYYY-MM-DD` — принудительно задать начало диапазона
- `--dry-run` — всё делает, но не пишет файлы и не шлёт в TG
- `--no-tg` — делает всё, но не шлёт дайджест

**При ошибке:** если любой шаг упал — короткое сообщение в TG с причиной, `git status` оставляем как есть, пользователь решает.

### 5.2 `/training:weekly` — review + план следующей недели

**Назначение:** воскресенье вечером. Два действия в одной команде, потому что review без плана бесполезен.

**Что делает:**

1. Убеждается что текущая неделя засинкана до воскресенья (при необходимости запускает те же шаги что `/training:sync`).
2. **Review прошлой недели:**
   - Читает `weeks/YYYY-WW.md`, все снапшоты недели, `me.md`, `goals.md`, `gym-program.md`, `weight-protocol.md`.
   - Анализирует:
     - Что выполнено vs план (по таблице)
     - Объёмы бег/зал vs цель
     - Z1+Z2% (цель 80)
     - CTL/ATL/TSB тренд
     - Сон/HRV/RHR средние и аномалии
     - Вес: тренд недели (min/max/avg), не отдельные дни
     - Сработавшие правила адаптации
     - Флаги тела
     - Прогрессия зала (факт vs «цели недели»)
   - Заполняет секцию «Review» в `weeks/YYYY-WW.md`.
   - Обновляет «Прогрессия ключевых лифтов» в `training/gym.md` если были новые максимумы.
3. **План следующей недели:**
   - Создаёт `weeks/YYYY-(WW+1).md` из шаблона.
   - Заполняет колонки «День» и «План» с учётом:
     - Периодизации из `goals.md` (тапер, операция, новые цели)
     - Правил адаптации (CTL/ATL, восстановление)
     - Прогрессии зала (веса на следующей неделе)
     - Индивидуальных ограничений (колено, плечо)
   - Предлагает раздел «Правила адаптации на этой неделе» (часто одинаковые, но корректируются)
   - Заполняет «Прогрессия зала — цели недели».
   - **Оставляет место для пользователя:** колонка «Заметка» в плане, «Open questions».
4. Архивирует прошлую неделю: `git mv weeks/YYYY-WW.md weeks/archive/YYYY-WW.md` (после того как пользователь подтвердит review).
5. Обновляет инлайн-блок текущей недели в `README.md`.
6. TG-дайджест «обзор недели» (см. §6).
7. `git commit -m "weekly: review WW + plan WW+1"` → `git push`.

**Интерактивность:** после того как скрипт предлагает Review и черновик плана — **пауза**, пользователь проверяет, правит, подтверждает. Только потом архивирование + push.

### 5.3 scripts/pipeline.py (новый)

Тонкий оркестратор для той части `/training:sync`, которую можно сделать без Claude-контекста (только Python + HTTP):

```
pipeline.py sync-range --from YYYY-MM-DD --to YYYY-MM-DD
pipeline.py build-week-summary --week YYYY-WW
pipeline.py digest --week YYYY-WW
```

MCP-вызовы к Garmin по-прежнему делает Claude внутри slash-команды (Python не может обращаться к MCP). То есть:
- `pipeline.py sync-range` делает intervals.icu + нормализацию
- slash-команда отдельно делает GetFast MCP и записывает в `data/raw/garmin/`
- затем slash-команда зовёт `pipeline.py build-week-summary` и `pipeline.py digest`

### 5.4 Существующие скрипты

- `fetch_intervals.py` — остаётся, получает `--date` аргумент (частично уже есть в roadmap `docs/repo-improvements-2026-04-23.md`).
- `normalize.py` — остаётся, получает `--date`.
- `send_telegram.py` — остаётся, получает `--file` для передачи готового дайджеста.

### 6.1 Sync-дайджест (после `/training:sync`)

Сон ср: 7.1 ч | HRV: 48→52 | RHR: 50→49

### 6.2 Weekly-дайджест (после `/training:weekly`)

Цель: «итоги недели + что на следующей». До ~35 строк.

```

НЕДЕЛЯ 21
Бег: 52 км (цель 50–55) ✅ | Z1+Z2 82%
Зал: 3/3 сессии ✅ | прогрессия: присед 100, жим +2.5

CTL 34 / ATL 55 / TSB −21
Вес тренд: 93.4→92.8 (−0.6 кг)

ЧТО ПОЛУЧИЛОСЬ
- интервалы в целевом темпе (4:00)
- паркран 21:30 — прогресс vs 04.04
- восстановление между сессиями

КОРРЕКТИРОВКИ НА НЕД 22
- тапер: бег 20–25 км до ср включительно
- последний зал A пн (интенсивно, не перегружать)
- операция 30.05 → зал и бег стоп с чт

НЕДЕЛЯ 22 (27.05–02.06) — ОПЕРАЦИОННАЯ
Пн Зал A · Вт интервалы 4×1500 · Ср easy 6–8 · Чт–Вс ⛔
```

## 9. Open questions (на следующий раунд)

Нерешённые детали, которые можно нащупать в ходе имплементации:

1. **Инлайн недели в README:** ручное копирование через `pipeline.py rebuild-readme` vs GitHub Actions. Старт с ручного, Actions — позже если понадобится.
2. **Архивация старых недель:** автоматом в `/training:weekly` или руками. Предложено авто, но можно спросить «архивировать?» перед `git mv`.
3. **Идемпотентность TG:** проверять, не отправлен ли уже дайджест за эту пару (date, mode). Нужно ли для `/training:sync` (может запускаться несколько раз за день при catchup). Предложение — хранить `data/logs/tg-sent.json` с хешами.
4. **Прогрессия ключевых лифтов:** 6 лифтов × N недель — такая таблица быстро выходит за ширину экрана после 10+ недель. Альтернатива — плейн-лист «Неделя 21: присед 100×10, РДЛ 90×8, …» и отдельный сводный график (out of scope).
5. **Один TG-бот и несколько окружений:** если будет dev/test запуск скриптов, нужен флаг `--chat-id override` или отдельный тест-бот.
